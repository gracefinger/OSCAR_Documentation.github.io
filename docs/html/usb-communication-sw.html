USB Communication

Setup for USB Communication

Overview

The desktop application is built with Electron, and USB communication is handled by a C module linked into the application. Because of this architecture, several libraries must be installed. The required libraries include libserialport (maintained by sigrok) and Node-API (node_api). Both are required for integration within the Electron project environment. However, only libserialport is required to compile the standalone C module independently of Electron.

Installing libserialport

The first step is to follow the installation instructions here

After following those instructions, build libserialport using a Windows-compatible toolchain such as MinGW, MSYS2, Cygwin, or MSVC Build Tools. After building, you should be able to locate the following source files inside the libserialport directory:

libs/libserialport/serialport.c
libs/libserialport/windows.c
libs/libserialport/timing.c

Make note of their file paths, as they will be referenced later in binding.gyp.

Installing node-gyp

Next, inside the Electron project (where Node has already been initialized with package.json), install node-gyp globally:

$ npm install -g node-gyp

Verify the installation using:

$ node-gyp --version

node-gyp requires Python. Confirm whether Python is installed or not:

$ python --version

If Python is not installed, download it from here.
You must also have the MSVC C++ Build Tools installed for Windows compilation.

After setup, your project structure should resemble:

project/
├── binding.gyp
├── src/
│  └── addon.c
└── package.json

Configuring binding.gyp

Open the binding.gyp file. In this file, you will manually map libserialport and your C module.

target_name defines the name of the generated .node file. It does not need to match the C source file name, but it must be referenced consistently throughout the project.

Replace all file paths below with the actual paths on your system:

{
  "targets": [
    {
    "target_name": "serial_addon",
    "sources": [
      "./Communication/USBCommunication.c",
      "C:/Users/catwo/dev/libs/libserialport/serialport.c",
      "C:/Users/catwo/dev/libs/libserialport/windows.c",
      "C:/Users/catwo/dev/libs/libserialport/timing.c"
    ],
    "include_dirs": [
      "C:/Users/catwo/dev/libs/libserialport"
    ],
    "libraries": [
      "setupapi.lib",
      "cfgmgr32.lib"
    ],
    "defines": [
      "LIBSERIALPORT_MSBUILD",
      "WIN32",
      "_WINDOWS"
    ],
    "msvs_settings": {
      "VCCLCompilerTool": {
        "ExceptionHandling": 1,
        "AdditionalOptions": [ "/wd4068" ]
      }
    }
  }
  ]
}


Important Notes:

  • include_dirs must point to the directory containing libserialport.h.
  • library links require Windows system libraries.
  • The libserialport source files are compiled directly into the addon.

Building the Native Module

Navigate to the Electron project root folder and run:

$ node-gyp configure
$ node-gyp build

If errors occur, resolve them before proceeding. To clean the build:

$ node-gyp clean

Successful compilation will generate:

build/Release/serial_addon.node

Rebuilding for Electron

Electron uses its own Node runtime version, so the module must be rebuilt against Electron headers.

To rebuild run:

$ npm install --save-dev electron-rebuild

Then run:

$ npx electron-rebuild

This ensures compatibility between Electron and the compiled native module.

Loading the Native Module in Electron

Create a file named preload.js and add the following code:

const { contextBridge } = require('electron');
const addon = require('./build/Release/serial_addon.node');

contextBridge.exposeInMainWorld("backendAPI", {
  openPort: addon.openPort
});

Important Notes:

  • Ensure the require path matches your generated .node file.
  • Use addon.openPort, not serial_addon.openPort, since addon is the imported object.

Data Streaming Setup

Data streaming within the Desktop Application is managed by the native C backend, which interfaces directly with the flight controller over USB serial communication. Unlike standard request–response interactions, streaming introduces a continuous data flow that is handled through a dedicated architecture involving dual COM ports, multithreading, and buffered data transfer to the frontend.

COM Port Identification & Differentiation

The application dynamically identifies and verifies the correct COM ports using a handshake-based detection mechanism. The port used for data streaming is separate from the command port used for standard requests.

The streaming COM port is dedicated exclusively to continuous telemetry data. It is discovered using the same process as the command port, but with a different handshake command. This allows the system to reliably distinguish between command and streaming channels without hardcoding port names.

Triggering Streaming Start & Stop

Streaming is initiated using the same command infrastructure as standard requests:

  • A start streaming command is sent through the command port
  • The firmware acknowledges the request and begins transmitting data on the streaming port
  • A stop command is issued to terminate streaming

This design keeps the streaming control flow consistent with the rest of the communication system and reuses existing packet structures and validation logic.

Multi-Threading Setup

To prevent blocking the main application flow and other commands, streaming is handled on a separate thread using the Windows API.

  • A dedicated streaming thread is created when streaming begins
  • The thread continuously reads incoming packets from the streaming port
  • Mutexes are used to safely manage shared resources such as the port handle and data queue

This allows real-time data to be processed without interfering with command execution or frontend responsiveness.

Forwarding Data to Frontend

Incoming streaming data is processed and transferred to the frontend through a queue-based system.

  • The streaming thread acts as a producer, pushing validated JSON packets into a circular buffer
  • The frontend interacts with the backend through exposed functions (via Node-API)
  • A polling function retrieves the next available packet from the queue

This producer–consumer model provides:

  • Thread-safe data transfer
  • Decoupling between data acquisition and UI updates
  • Smooth handling of continuous telemetry streams
