USB Communication

Command Setup

The USB communication is set up to use standard UART protocol. Communications have 1 start bit, 1 stop bit, no parity, 115200 baud rate, and 8 data bits. The PC and flight controller MCU are set up in a master slave configuration. Commands are sent from the master, the PC, to the slave, flight controller. Responses are expected to be sent back from the slave device. Commands are formatted with a command header to identify the command followed by any necessary data and a CRC redundancy check. The command header is 2 bytes and formatted as a hex number. The number will always start with an 0x8 followed by 3 more hex numbers to determine the ID of the command. Responses are formatted similarly where that is a 2 byte header dictating what the response is for. The CRC is a 16 bit (2 byte) calculation performed on the data being loaded into the packet. See CRC section for more details on the calculation. The maximum size of the transmission is 64 bytes. The following illustration shows the format in which a command is to be sent.



Commands sent from the PC should be in the form of a standard array. The flight controller MCU will extract the first two bytes of the transmission to determine what the response needs to be. Any data that follows should be unique to the command. To prevent parsing errors updating data values should be done inside individual commands. For example, updating a variable P should be sent in a command specific to updating that variable P. It is not recommended to send data to update both variables P and X in a single command.

Before the command is fully processed, the CRC is checked. Upon failure, a CRC failure response header is sent back to the host PC. The CRC is checked by running the full packet through the calculation and comparing it with the CRC that was sent along with the data. The CRC calculator excludes the CRC data that was sent. In this case, the length of the buffer being calculated needs two bytes subtracted from it.

Data is sent through the usb in packets of up to 64 bytes. If the data or buffer being loaded is more than 64 bytes it will need to be split up into multiple packets. The line coding is set to use UART so the device behaves like a USB to serial device for the purposes of the host machine receiving and sending back data.

Response Setup

Once the fight controller MCU receives a command, it will parse through the input and either retrieve or update values. When it is ready to send a response, the flight controller code will build a JSON object to simplify data interpretation on the software side. As mentioned earlier, the response communications also have a command header. These will start with 0x5, followed by 3 more hex bits to identify the command they respond to. These headers are also 2 bytes in size. Following the header is the length of the packet being sent back to the host computer. This length is the size of the data and the CRC. It does not include the command header bytes nor the packet length byte.



To setup JSON responses we are using the cJSON library found on the GitHub page linked here.

Multiple library calls are required to correctly build the JSON object. The following examples are derived from the GitHub ReadMe for easier access:

Building JSON in C

//NOTE: Returns a heap allocated string, you are required to free it after use.
char *create_monitor_with_helpers(void)
{
  const unsigned int resolution_numbers[3][2] = {
    {1280, 720},
    {1920, 1080},
    {3840, 2160}
  };
  char *string = NULL;
  cJSON *resolutions = NULL;
  size_t index = 0;

  cJSON *monitor = cJSON_CreateObject();

  if (cJSON_AddStringToObject(monitor, "name", "Awesome 4K") == NULL)
  {
    goto end;
  }

  resolutions = cJSON_AddArrayToObject(monitor, "resolutions");
  if (resolutions == NULL)
  {
    goto end;
  }

  for (index = 0; index < (sizeof(resolution_numbers) / (2 * sizeof(int))); ++index)
  {
    cJSON *resolution = cJSON_CreateObject();

    if (cJSON_AddNumberToObject(resolution, "width", resolution_numbers[index][0]) == NULL)
    {
      goto end;
    }

    if (cJSON_AddNumberToObject(resolution, "height", resolution_numbers[index][1]) == NULL)
    {
      goto end;
    }

    cJSON_AddItemToArray(resolutions, resolution);
  }

  string = cJSON_Print(monitor);
  if (string == NULL)
  {
    fprintf(stderr, "Failed to print monitor.\n");
  }

end:
  cJSON_Delete(monitor);
  return string;
}

When creating a JSON object it is important to add the elements to the object that you are needing to transmit. This would be the important data Software needs to display. Make sure to communicate how your JSON is formatted with the software team so that it can be parsed correctly. Moreover, creating JSON objects takes allocated memory. Be sure to delete the JSON object after transmitting it over USB. Before the JSON object can be transmitted it needs to be transferred into a standard string. The cJSON_Print function should be able to handle this.

ResponseJSON = cJSON_CreateObject();

//Create a JSON object for all the numbers to be used

CRC Calculation

The CRC calculator follows the standard crc16_ccitt format. It takes in a buffer and its length. Each element in the buffer is xor'ed with the polynomial. The polynomial used is as follows:


x16 + x12 + x5 + 1

It is used in the hexadecimal form 0x1021u. The CRC is formatted in a 16 bit integer.
