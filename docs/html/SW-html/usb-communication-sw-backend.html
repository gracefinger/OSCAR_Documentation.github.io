    Backend USB Communication

    Data Flow

    Overview

    This section will provide information about the USB communication and data flow. For actual implementation steps involving JavScript and Electron, check out the Frontend USB Communication section. This Data Flow section, however, will be the same across both pages. This document will focus more on the C side of things. We will start by going over the overall flow of the data.

    Communication Flow & Diagram

    The USB communication system consists of three major layers:

    • Desktop Application (Electron / JavaScript)
    • Native Communication Layer (C module using libserialport)
    • Flight Controller Firmware (MCU)

    Each layer has a defined role in the communication pipeline. Commands originate from the desktop application, are transmitted through the native C module using UART over USB, processed by the firmware, and then a response is returned following the same path in reverse.

    The communication flow ensures that high-level application logic remains separated from low-level serial communication.

    Command Transmission Flow (PC to Flight Controller)

    1. User Action in Desktop Application: A user interacts with the Electron UI (for example clicking a button to request battery information).
    2. JavaScript Calls Native Function: The JavaScript code calls an exposed API function through the Electron contextBridge.
    3. Node-API Bridge: The call is forwarded through Node-API to the compiled native module (serial_addon.node).
    4. Native C Module Builds Command: The native C code constructs the command packet:
      • [CMD Header (0x800X)] + [Command Data]
      • The packet is prepared as a byte array before transmission.
    5. USB UART Transmission: The C module sends the command through libserialport using the configured UART settings:
      • 115200 baud
      • 8 data bits
      • no parity
      • 1 stop bit
    6. Firmware Receives Command: The flight controller firmware reads the incoming bytes from the UART buffer.
    7. Command Parsing: The firmware extracts the first 2 bytes to determine the command ID.
      • Example: 0x8009 → e_GetInitialValuesVolt
    8. Command Execution: The firmware processes the command by either:
      • retrieving stored values
      • updating configuration parameters
      • requesting sensor data

    Response Transmission Flow (Flight Controller to PC)

    1. Response Construction: After processing the command, the firmware constructs a response packet containing:
      • [Response Header (0x500X)] + [JSON Data]
    2. JSON Creation: The firmware uses the cJSON library to construct a JSON object containing labeled data values.
    3. Serialize JSON: The JSON object is converted to a string using:
      • cJSON_Print()
    4. Transmit Response: The response packet is transmitted back over USB UART to the PC.
    5. Native C Module Receives Data: The C module reads the incoming serial buffer using libserialport.
    6. Response Verification: The first two bytes are checked to confirm the response header.
    7. Pass Data to JavaScript: The received JSON string is forwarded through the Node-API layer to the Electron application.
    8. Application Parses JSON: The Electron application parses the JSON and updates the UI accordingly.

    Native C Module

    Overview

    The native C module serves as the communication bridge between the Electron application and the flight controller hardware. It is responsible for:

    • Constructing command packets
    • Handling USB UART communication
    • Verifying responses (CRC, headers, length)
    • Returning structured JSON data to JavaScript
    • Triggering and forwarding data streaming

    This module is compiled as a Node.js addon using Node-API (N-API), allowing it to be called directly from JavaScript.

    Streaming Data

    Overview

    In addition to standard request-response communication, the OSCAR Ground Station supports continuous data streaming from the flight controller. This is primarily used for real-time telemetry, such as gyroscope orientation data, which must be updated frequently without requiring repeated manual requests from the user.

    Unlike discrete commands (such as populateAll or updateValue), streaming operates continuously in the background and pushes updates to the frontend as new data becomes available.

    Streaming Flow

    The streaming pipeline is structured as follows:

    1. Renderer (UI) →
    2. ipcRenderer →
    3. Main Process →
    4. ipcMain →
    5. Worker Thread (Command Controller) →
    6. Streaming Thread (continuous data handler) →
    7. Native C Module (continuous serial read via libserialport) →
    8. Flight Controller

    The native module maintains an active serial connection and continuously reads incoming telemetry data. This data is forwarded through the streaming thread, which parses and routes updates back through the Electron IPC system to the renderer process for UI updates.

    Backend

    The desktop application is built with Electron, and USB communication is handled by a C module linked into the application. Because of this architecture, several libraries must be installed. The required libraries include libserialport (maintained by sigrok) and Node-API (node_api). Both are required for integration within the Electron project environment. However, only libserialport is required to compile the standalone C module independently of Electron.

    Installing libserialport

    The first step is to follow the installation instructions here

    After following those instructions, build libserialport using a Windows-compatible toolchain such as MinGW, MSYS2, Cygwin, or MSVC Build Tools. After building, you should be able to locate the following source files inside the libserialport directory:

    libs/libserialport/serialport.c
    libs/libserialport/windows.c
    libs/libserialport/timing.c

    Make note of their file paths, as they will be referenced later in binding.gyp.

    Installing node-gyp

    Next, inside the Electron project (where Node has already been initialized with package.json), install node-gyp globally:

    $ npm install -g node-gyp

    Verify the installation using:

    $ node-gyp --version

    node-gyp requires Python. Confirm whether Python is installed or not:

    $ python --version

    If Python is not installed, download it from here.
    You must also have the MSVC C++ Build Tools installed for Windows compilation.

    After setup, your project structure should resemble:

    project/
    ├── binding.gyp
    ├── src/
    │  └── addon.c
    └── package.json

    Building the Native Module

    Navigate to the Electron project root folder and run:

    $ node-gyp configure
    $ node-gyp build

    If errors occur, resolve them before proceeding. To clean the build:

    $ node-gyp clean

    Successful compilation will generate:

    build/Release/serial_addon.node

    Rebuilding for Electron

    Electron uses its own Node runtime version, so the module must be rebuilt against Electron headers.

    To rebuild run:

    $ npm install --save-dev electron-rebuild

    Then run:

    $ npx electron-rebuild

    This ensures compatibility between Electron and the compiled native module.

    Loading the Native Module in Electron

    Create a file named preload.js and add the following code:

    const { contextBridge } = require('electron');
    const addon = require('./build/Release/serial_addon.node');

    contextBridge.exposeInMainWorld("backendAPI", {
      openPort: addon.openPort
    });

    Important Notes:

    • Ensure the require path matches your generated .node file.
    • Use addon.openPort, not serial_addon.openPort, since addon is the imported object.

    Native C Module - Backend

    The native C module handles low-level USB communication between the Electron app and the flight controller firmware. It exposes a simple API to JavaScript while managing serial communication details. Responsibilities include:

    • Maps frontend command strings to protocol headers (e.g., "GetInitialValuesVolt" → 0x8009).
    • Verifies and opens COM ports with a handshake (0x8000 → 0x5000).
    • Constructing and sending packets with a header, length, payload, and CRC.
    • Reading and validating responses using length fields and CRC checks.
    • Returns JSON responses or structured error messages to the frontend.
    • Handles errors such as invalid commands, CRC failures, port failures, timeouts, and memory allocation issues.

    The module is structured into the following layers:

    1. USB Communication Layer (libserialport)
      • Handles opening/closing ports, configuring baud rate, parity, stop bits, and reading/writing bytes.
      • Provides blocking reads/writes with timeout handling.
      • check() function is used to evaluate libserialport return codes for errors.
    2. Command & Packet Layer
      • Command Mapping
        • Maps frontend command strings to protocol headers using map_command().
        • Provides helper functions for sending commands with or without payloads.
        • If a command is invalid it returns: 0xFFFF.
      • Packet Structure
        • All communication follows the command packet setup listed on Frontend USB Communication
      • CRC Generation & Validation
        • Before sending a packet:
          • A CRC value is computed over the header, length, and payload.
          • The CRC is appended to the end of the packet.
        • After receiving a response:
          • The CRC is recomputed from the received data.
          • The computed CRC value is compared to the received CRC value.
        • If the CRC does not match:
          • The packet is rejected.
          • The request either retries or fails with an error
      • Length-Based Reads
        • Instead of relying solely on timeouts, the module uses the length field to determine how many bytes to read:
          • Read initial bytes (header + length)
          • Extract the expected message size
          • Continue reading until the full packet is received
        • This improves:
          • Performance (avoids waiting for full timeout)
          • Reliability (ensures complete packet reads)
    3. Application Interface (Node-API)
      • Exposes C functions to JavaScript via Electron’s contextBridge.
      • Functions include populateAll() for reading an entire page and updateValue() for writing individual values.
      • Returns JSON strings or error messages back to the frontend.
    4. Verification & Handshake
        • find_com_port() iterates over all serial ports and performs handshake validation with firmware (0x8000 → 0x5000).
        • Stores verified port in verified_port for subsequent communication.

    Functions exposed to the frontend include:

    • populateAll(command)
      • Reads all page values from the device; returns JSON or error.
    • updateValue(command, payload)
      • Sends a single value update; returns JSON or error.
    • startStreaming(command)
      • Starts streaming data for specified page (command); returns 1 on success, 0 on failure.
    • stopStreaming()
      • Signals to the streaming loop that it should stop. Triggers the stop streaming request to be sent to firmware; returns 1 on success, 0 on failure.
    • GetNextStreamPacket()
      • Internal backend function implemented in the C module that retrieves the next available data packet from a thread-safe queue. This function does not directly communicate with the frontend; instead, it is repeatedly called by the backend polling loop, and any returned data is then forwarded to the frontend through streamData events.

    Note: There are many other functions in the C module. These are just the ones that the frontend Electron project has access to.

    The module uses Node-API (napi) to bridge between C and JavaScript:

    • PopulateAll
      • calls populateAll() and converts the C string to a JavaScript string and vice versa.
    • UpdateValue
      • calls updateValue() and converts the C string to a JavaScript string and vice versa.
    • StartStreaming
      • calls startStreaming and converts the C string to JavaScript string and vice versa
    • StopStreaming
      • calls stopStreaming.
    • GetNextStreamPacket
      • calls getNextStreamPacket and allows forwarding of packets to frontend.
    • All wrappers handle NULL values and memory allocation errors safely.

    This allows JavaScript code to call the native functions as if they were regular async functions, while keeping the low-level details encapsulated.

    Error Handling & Recovery - Backend

    The USB communication system includes error detection and recovery mechanisms at multiple layers. These mechanisms ensure that failures such as missing devices, invalid responses, or serial communication errors do not crash the desktop application and instead return structured error messages that can be handled by the Electron UI.

    Errors are handled in three stages:

    1. Serial communication errors (libserialport layer)
    2. Device verification failures
    3. Application-level protocol errors

    The native C module uses libserialport for all UART communication. Most configuration calls return a value of type sp_return, which indicates whether the operation succeeded. To simplify error handling, there is a helper function used. This function is int check(enum sp_return result). Upon error, this function prints an error message and aborts execution. Possible error conditions include:

    • SP_ERR_ARG - Invalid argument passed to libserialport function
    • SP_ERR_FAIL - Generic failure reported by the driver
    • SP_ERR_SUPP - Operation not supported
    • SP_ERR_MEM - Memory allocation failure

    Device Detection & Verification

    Before any commands are transmitted, the application verifies that the correct USB device is connected. This process is handled by the function: find_com_port().

    The function performs the following steps:

    1. Enumerates all available serial ports using sp_list_ports().
    2. Attempts to open each port.
    3. Sends a handshake command (0x8000).
    4. Waits for the expected acknowledgment 0x5000 from the firmware.
    5. If the correct response is received, the port name is stored in verified_port.

    If no device responds with the expected handshake, the function returns failure. The application then returns the following JSON error to the Electron frontend allowing the UI to display an error message.

    Command Validation & Response Verification

    Before sending a command to the firmware, the command string provided by the JavaScript layer is translated into a command header using: map_command(). If the command does not exist in the mapping table, then the function returns 0xFFFF to signify an error occurred.

    After sending a command packet, the system waits for a response from the firmware. All valid responses must contain a header beginning with: 0x50XX. The header is checked using: (response_header & 0xFF00) != 0x5000 and response_header == 0xFFFE. If the response header does not match this format, the message is considered invalid and the request fails. This prevents corrupted or unrelated serial data from being interpreted as valid JSON.

    Length & CRC Validation

    In addition to header validation, all packets are now validated using both a length field and a CRC value.

    After receiving the initial portion of a packet (header and length):

    • The length field is used to determine the expected total size of the remaining data
    • The system continues reading until the full packet is received

    This prevents incomplete reads and reduces reliance on timeout-based detection.

    Once the full packet is received:

    • A CRC value is computed over the received data (excluding the received CRC field)
    • The computed CRC is compared against the CRC included in the packet
      • If:
        • The received data length does not match the expected length, or
        • The CRC values do not match

    Then the packet is considered invalid and is discarded. This ensures that:

    • Corrupted packets are not processed
    • Partial transmissions are rejected
    • Only fully valid responses are passed to higher layers

    Timeout Handling

    The communication layer uses blocking reads with a timeout: sp_blocking_read(..., timeout_ms). For command responses, the timeout is set to 2000 ms. Possible outcomes include from this timeout function include:

    • bytes_read > 0: Response received successfully
    • bytes_read == 0: Timeout occurred (no response)
    • bytes_read < 0: Serial read error

    Timeouts typically indicate:

    • device disconnection
    • firmware crash
    • incorrect COM port selection

    Note: Timeouts are primarily used as a fallback method to determine message completion or error. The length-based packet handling is the primary method.

    All dynamic memory allocations are checked before use.If memory allocation fails, the system:

    1. Closes the serial port
    2. Frees allocated resources
    3. Returns NULL to the caller

    Error Propogation to the Frontend

    Errors generated by the native module are returned to the JavaScript layer as JSON error objects. EX: { "ERROR": "error description" }. The Electron renderer checks for the presence of the ERROR field: If an error is present, a modal dialog is displayed to the user. This approach allows the native module to communicate failures without crashing the desktop application.

    Data Streaming Setup

    Data streaming within the Desktop Application is managed by the native C backend, which interfaces directly with the flight controller over USB serial communication. Unlike standard request–response interactions, streaming introduces a continuous data flow that is handled through a dedicated architecture involving dual COM ports, multithreading, and buffered data transfer to the frontend.

    COM Port Identification & Differentiation

    The application dynamically identifies and verifies the correct COM ports using a handshake-based detection mechanism. The port used for data streaming is separate from the command port used for standard requests.

    The streaming COM port is dedicated exclusively to continuous telemetry data. It is discovered using the same process as the command port, but with a different handshake command. This allows the system to reliably distinguish between command and streaming channels without hardcoding port names.

    Triggering Streaming Start & Stop

    Streaming is initiated using the same command infrastructure as standard requests:

    • A start streaming command is sent through the command port
    • The firmware acknowledges the request and begins transmitting data on the streaming port
    • A stop command is issued to terminate streaming

    This design keeps the streaming control flow consistent with the rest of the communication system and reuses existing packet structures and validation logic.

    Multi-Threading Setup

    To prevent blocking the main application flow and other commands, streaming is handled on a separate thread using the Windows API.

    • A dedicated streaming thread is created when streaming begins
    • The thread continuously reads incoming packets from the streaming port
    • Mutexes are used to safely manage shared resources such as the port handle and data queue

    This allows real-time data to be processed without interfering with command execution or frontend responsiveness.

    Forwarding Data to Frontend

    Incoming streaming data is processed and transferred to the frontend through a queue-based system.

    • The streaming thread acts as a producer, pushing validated JSON packets into a circular buffer
    • The frontend interacts with the backend through exposed functions (via Node-API)
    • A polling function retrieves the next available packet from the queue

    This producer–consumer model provides:

    • Thread-safe data transfer
    • Decoupling between data acquisition and UI updates
    • Smooth handling of continuous telemetry streams
